**"What is RAG? Have you implemented it?"**

"Yeah, I've built it twice so let me explain from how I think about it rather than the textbook definition.

The core problem RAG solves is this: an LLM knows what it learned during training but it has no idea what's in your company's internal docs, your emails, your codebase. You can't fine-tune it every time something changes. RAG says — okay, at the time the user asks something, go retrieve the relevant information and hand it to the model as context. The model then answers based on what it just received, not just what it memorised.

The pipeline has three parts. Indexing: you take your documents, break them into chunks, convert each chunk to a vector embedding using an embedding model, store those in a vector database. Retrieval: when someone asks a question, you embed that question too, then find chunks that are semantically closest to it using cosine similarity. Generation: you inject those chunks into the prompt and let the LLM generate a grounded answer.

In Mnemo — my memory API — I added two layers on top of vanilla RAG. One is decay scoring: recent memories get a higher weight even if an older memory has slightly better cosine similarity. Because in most real use cases, what you said last week matters more than what you said six months ago. The second is contradiction detection: if a new memory conflicts with an existing one, the system flags it instead of silently storing both and confusing the model later.

So yes, I've implemented it, and I've thought beyond the basic version."

**"Explain a Transformer to me"**

"So before transformers we had RNNs, and the fundamental problem with RNNs was they processed sequences token by token, in order. Which means if you're reading a sentence and the important context is 50 words back, by the time you get to it the earlier information has basically faded out. The network forgets.

The transformer's insight was: what if every word can look at every other word at the same time? That's what attention is. Each token asks 'how relevant is every other token to me right now?' and you get a weighted sum. You're not reading left to right — you're looking at the whole thing at once.

Multi-head attention runs this process multiple times in parallel — each head learns to focus on different kinds of relationships. One head might focus on syntactic structure, another on semantic meaning. You run 8 or 12 heads, concatenate the outputs, and you've captured richer relationships than any single attention head could.

Then there's positional encoding because the attention mechanism itself has no concept of order. You add a position vector to each token embedding to tell the model where in the sequence each token sits.

The encoder builds a contextual representation of the input. The decoder generates the output, and it uses cross-attention to look at the encoder's representations while it's generating each output token.

BERT uses just the encoder — it was trained on masked language modelling, so it learned to predict missing words. GPT uses just the decoder, trained to predict the next token. For most chat applications you're working with GPT-style decoder-only architectures."

**"What's the difference between fine-tuning, RAG, and prompt engineering?"**

"They're three different levers and they're not interchangeable — each one solves a different problem.

Prompt engineering is the cheapest and fastest. You don't touch the model at all. You just craft your input more carefully — add examples, structure your request, break complex tasks into steps. Zero-shot just tells the model what to do. Few-shot gives it 2–3 examples of the format you want. Chain-of-thought asks it to reason step by step before answering. Works well when the model already has the knowledge and you just need to steer the output.

RAG is for when the model doesn't have the knowledge — or the knowledge changes frequently. You're not changing the model, you're giving it fresh information at the time of each query. I use this in Auren for email context and in Mnemo for persistent memory. The limitation is that retrieval quality matters a lot — if you retrieve the wrong chunks, the model will still sound confident while being wrong.

Fine-tuning actually changes the model's weights using your labelled dataset. This is for when you need the model to write in a specific tone consistently, follow a very specific output format every time, or deeply understand a domain-specific vocabulary. It's expensive, you need GPU access, and you need a good dataset. For most startup use cases RAG is the right call. Fine-tuning makes sense after you've exhausted the other two and you have clear evidence they're not sufficient."

**"Explain OOPs. Can you give me examples?"**

"Sure. OOPs is basically a way to organise code around objects instead of just functions running in sequence. The four pillars —

Encapsulation is about bundling data and the functions that operate on it inside one unit, and deciding what the outside world gets to see. In Python you signal private stuff with an underscore prefix — it's convention, not hard enforcement. The benefit is you can change how something works internally without breaking everything that uses it.

Inheritance is when one class gets the attributes and methods of another. Saves you from rewriting the same logic. Python supports multiple inheritance — one class can inherit from multiple parents — and uses something called MRO, Method Resolution Order, to figure out which parent's version of a method gets called when there's a conflict. It follows C3 linearisation.

Polymorphism is the same method name doing different things depending on what object you're calling it on. In Python this is largely duck typing — if the object has the method, it works. You don't need explicit interfaces like in Java.

Abstraction is hiding the implementation and just exposing what the user of your class needs. Python has an ABC module — Abstract Base Classes — where you can define methods that every subclass must implement. If a subclass doesn't implement a required abstract method, you get an error when you try to instantiate it."

**"What are ACID properties?"**

"ACID is what makes database transactions reliable, especially when multiple things are happening at the same time or something crashes mid-operation.

Atomicity is all or nothing. Classic example: you're transferring money between two bank accounts. The debit from account A and the credit to account B need to happen together. If the system crashes after the debit but before the credit, atomicity rolls the whole thing back. You don't end up with money disappearing.

Consistency means the database goes from one valid state to another. Whatever rules and constraints exist — foreign keys, unique constraints, check constraints — they all hold after every transaction.

Isolation is about concurrent transactions not interfering with each other. If two transactions are running at the same time, each one should see a consistent snapshot, not half-finished work from the other. Different isolation levels give you different trade-offs between correctness and performance.

Durability means once a transaction says it committed, it's committed. Even if the machine crashes immediately after, the data is persisted to disk. It's not just in memory."

**"What is normalisation?"**

"Normalisation is about organising your database so you're not storing the same information in multiple places, which causes problems when you try to update it.

First Normal Form just says every cell in your table should hold one value, not a list. If you're storing comma-separated values in a single column, that's a 1NF violation.

Second Normal Form builds on that by saying if you have a composite primary key, every non-key column needs to depend on the whole key, not just one part of it. Partial dependency is what it eliminates.

Third Normal Form eliminates transitive dependency — which means non-key columns shouldn't depend on other non-key columns. If knowing the zip code tells you the city, and the city isn't part of the primary key, that's a transitive dependency. You'd split that into its own table.

BCNF is slightly stricter than 3NF. It says every determinant — any column that uniquely determines another column — must be a candidate key.

In real production systems you normalise to 3NF and then sometimes deliberately denormalise specific tables where you need faster reads. It's a trade-off."

**"What is a deadlock?"**

"A deadlock is when two or more processes are stuck, each one waiting for a resource that another one is holding, and none of them can proceed because they're all waiting on each other.

Four conditions have to all be true at the same time for a deadlock to happen. First, mutual exclusion — at least one resource can only be used by one process at a time. Second, hold and wait — a process is holding a resource and also waiting to get more. Third, no preemption — you can't forcibly take a resource away from a process, it has to release it voluntarily. Fourth, circular wait — you have a cycle where process A is waiting for process B which is waiting for process C which is waiting for process A.

To prevent it you break one of those conditions. The most common approach is imposing a fixed ordering on resources — if all processes always request resources in the same order, you can never get a circular wait. The Banker's Algorithm is a more sophisticated approach where before granting a resource, the system checks whether the resulting state is still safe — meaning there's at least one sequence in which all processes can complete."

---

**"What's the difference between a process and a thread?"**

"A process is an independent running program with its own memory space — its own heap, stack, code, data. Processes are isolated. If one crashes, it doesn't take down others because they're not sharing memory.

A thread lives inside a process. Multiple threads within the same process share the heap and the code, but each thread has its own stack and its own registers. They're lighter than processes because you're not duplicating memory. Switching between threads is also cheaper than switching between processes.

The trade-off is that because threads share memory, you can get race conditions — two threads reading and writing the same data at the same time in an undefined order. That's why you need synchronization like mutexes or semaphores."
