✓ What are ACID properties?
50s DBMS
ACID stands for Atomicity, Consistency, Isolation, and Durability. These properties guarantee that database transactions are processed reliably: 1. **Atomicity:** 'All or nothing'. If any part of a transaction fails, the entire transaction is rolled back (e.g., in a bank transfer, deducting money from Account A and depositing in Account B must both succeed or both fail). 2. **Consistency:** A transaction must transition the database from one valid state to another, maintaining all schema constraints and rules. 3. **Isolation:** Concurrent execution of transactions leaves the database in the same state as if they were executed sequentially. 4. **Durability:** Once a transaction commits, its changes are permanently written to non-volatile storage (disk) and survive system crashes.
Key points: AtomicityConsistencyIsolationDurabilityAll or nothingCrash recovery
⚠️ Do NOT say: Just naming the terms without explaining a real-world example like bank transfer
✓ Explain database normalisation (1NF, 2NF, 3NF, BCNF) with examples.
60s DBMS
Normalisation is the process of organising data in a database to reduce redundancy and eliminate anomalies (Insert, Update, Delete): 1. **1NF (First Normal Form):** Table cells must contain atomic (indivisible) values, and there must be no repeating groups. * *Example:* If a `Student` table has a `Phone` column with multiple numbers separated by commas, it violates 1NF. 2. **2NF (Second Normal Form):** Must be in 1NF, and all non-key attributes must be fully functionally dependent on the primary key (no partial dependency). * *Example:* A table with `(StudentID, CourseID, CourseFee)` where `CourseFee` depends only on `CourseID` (part of composite key) violates 2NF. Split it. 3. **3NF (Third Normal Form):** Must be in 2NF, and there must be no transitive dependencies (non-key attributes depending on other non-key attributes). * *Example:* Table with `(StudentID, DeptID, DeptName)` where `DeptName` depends on `DeptID`, which depends on `StudentID`. Split it. 4. **BCNF (Boyce-Codd Normal Form):** A stronger version of 3NF. For any dependency `X -> Y`, `X` must be a superkey. * *Example:* A table with composite keys where a part of the candidate key depends on a non-prime attribute violates BCNF.
Key points: Redundancy reductionAnomalies preventionAtomic valuesPartial dependencyTransitive dependencyBCNF superkey rule
⚠️ Do NOT say: Mixing up the definitions of 2NF and 3NF, or failing to mention anomalies
✓ What is the difference between WHERE and HAVING?

✓ What is database indexing and why does it speed up queries?
40s DBMS
An index is a pointer structure (typically a B-Tree or B+ Tree) that the database engine uses to locate rows in a table without scanning the entire table (Sequential Scan). * **How it works:** Instead of checking every row (O(N) search), the database traverses a balanced tree structure in O(log N) time to find pointers to the physical rows on disk. * **Cost:** Indexing speeds up `SELECT` queries but slows down `INSERT`, `UPDATE`, and `DELETE` operations because the database must update the index tree structure every time the data changes. It also consumes additional disk space.
Key points: B-Tree/B+ TreeSequential Scan vs Index ScanO(log N) complexityDML overheadStorage cost

✓ What are database transactions? Explain isolation levels.
50s DBMS
A Transaction is a single logical unit of database work. To manage concurrency, SQL databases define 4 main isolation levels, which trade off speed for protection against read anomalies: 1. **Read Uncommitted:** Lowest level. Allows **Dirty Reads** (reading uncommitted data from other transactions that might rollback). 2. **Read Committed:** Prevents Dirty Reads. Only reads committed data. However, **Non-Repeatable Reads** are possible (rereading the same row within a transaction returns different values because another transaction modified and committed it). 3. **Repeatable Read:** Prevents Non-Repeatable Reads. Guarantees that rereading data returns the same values. However, **Phantom Reads** can occur (new rows inserted by other transactions appear in subsequent query runs). 4. **Serializable:** Highest level. Strict locking or ordering. Prevents all anomalies (Dirty, Non-Repeatable, Phantom Reads) but has massive performance overhead due to blocking.
Key points: Dirty ReadsNon-Repeatable ReadsPhantom ReadsRead CommittedRepeatable ReadSerializable overhead
⚠️ Do NOT say: Failing to explain what a 'dirty read' or 'phantom read' actually means

✓ Explain deadlock in DBMS and how to prevent it.
45s DBMS
A DBMS deadlock occurs when two or more transactions are waiting indefinitely for locks held by each other, creating a circular dependency. *Example:* Transaction A holds lock on Row 1 and wants Row 2. Transaction B holds lock on Row 2 and wants Row 1. **Prevention and Resolution:** 1. **Deadlock Detection:** The database runs a background process checking the "Wait-For Graph" for cycles. If found, it aborts and rolls back one transaction (the victim). 2. **Locking Order:** Ensure all transactions lock resources in a strict, identical sequence. 3. **Lock Timeouts:** Abort transactions that wait for a lock longer than a set threshold.
Key points: Wait-For GraphCircular dependencyDeadlock resolution / Aborting victimStrict locking orderLock timeout

✓ Explain the CAP Theorem.
45s DBMS
The CAP Theorem states that a distributed data store can simultaneously provide at most two of the following three guarantees: 1. **Consistency (C):** Every read receives the most recent write or an error. 2. **Availability (A):** Every non-failing node returns a non-error response (without guarantee that it contains the most recent write). 3. **Partition Tolerance (P):** The system continues to operate despite arbitrary message loss or node failures. Because physical networks are guaranteed to experience splits/failures, we **must** design for Partition Tolerance (P). Therefore, distributed databases must choose between: * **CP (Consistency/Partition Tolerance):** Reject writes if sync cannot be guaranteed (e.g., MongoDB, HBase). * **AP (Availability/Partition Tolerance):** Accept writes and resolve conflicts later, guaranteeing eventual consistency (e.g., Cassandra, DynamoDB).
Key points: Distributed databaseConsistency, Availability, Partition ToleranceMust choose PCP vs AP tradeoff
⚠️ Do NOT say: Conflating CAP consistency (distributed node sync) with ACID consistency (schema rules validation)

✓ Explain optimistic vs pessimistic locking.
45s DBMS
These are two strategies used to manage concurrent updates: * **Optimistic Locking:** * Assumes conflicts are rare. It does not lock rows. * Instead, it includes a version number or timestamp in the update query (e.g., `UPDATE Users SET salary = 5000, version = version + 1 WHERE id = 1 AND version = 5`). * If another transaction updated the row first, the version query updates 0 rows, triggering a rollback and retry in the application. Excellent for high-concurrency read-heavy setups. * **Pessimistic Locking:** * Assumes conflicts are common. It locks rows immediately upon query execution (e.g., `SELECT FOR UPDATE`). * Other transactions must block and wait until the locking transaction commits/rolls back. Prevents updates but reduces throughput.
Key points: Optimistic: version/timestamp checkPessimistic: locks rows / block writesRead-heavy vs Write-heavy setups
⚠️ Do NOT say: Recommending pessimistic locking for read-heavy systems where optimistic locking is much faster

✓ What is the difference between a process and a thread?
50s OS
The fundamental differences lie in resource isolation and context switching costs: * **Process:** * An active execution instance of a program. * Has its own isolated address space (code, data, stack, heap). * Inter-process communication (IPC) is required (e.g., sockets, pipes). * Heavyweight context switching (involves flushing the TLB and changing page tables). * *Analogy:* A completely separate browser application. * **Thread:** * The smallest unit of CPU scheduling. * Sibling threads share the parent process's memory space, code, and global variables, but maintain their own program counter and stack. * Shared memory communication; lightweight. * Very fast context switching (TLB remains warm). * *Analogy:* Multiple active tabs inside the same browser application.

✓ Explain deadlock and its four necessary conditions.
45s OS
A deadlock occurs when a set of processes are blocked indefinitely because each holds a resource and waits for another resource held by someone else in the cycle. **Four Coffman Conditions (Must hold simultaneously):** 1. **Mutual Exclusion:** At least one resource must be held in a non-shareable mode (only one process can use it at a time). 2. **Hold and Wait:** A process must be holding at least one resource and waiting to acquire additional resources held by other processes. 3. **No Preemption:** Resources cannot be forcibly taken from a process; they must be released voluntarily. 4. **Circular Wait:** A closed loop of processes exists where each process holds resources required by the next (e.g., P0 waits for P1, P1 waits for P0).
Key points: Mutual ExclusionHold and WaitNo PreemptionCircular WaitAll 4 must hold simultaneously
⚠️ Do NOT say: Listing deadlock prevention methods instead of the 4 conditions themselves
✓ What is virtual memory? How does paging work?
50s OS
Virtual memory is a memory management technique that gives programs the illusion of a large, continuous block of main memory (RAM), even if it is fragmented or exceeds physical memory limits. **How Paging Works:** * The logical address space is divided into fixed-size blocks called **Pages**. * Physical memory is divided into identical fixed-size blocks called **Frames**. * The CPU generates a **Logical Address** consisting of a Page Number (`p`) and Page Offset (`d`). * The hardware (MMU) uses a **Page Table** to map the virtual Page Number `p` to the physical Frame Number `f`. The physical address becomes `f` concatenated with offset `d`. * To speed this up, a hardware cache called the **Translation Lookaside Buffer (TLB)** stores recent virtual-to-physical address mappings.
Key points: Virtual-to-Physical translationPages and FramesMemory Management Unit (MMU)Page Table indexingTLB lookup
⚠️ Do NOT say: Saying pages and frames are of different sizes (they must be identical)
✓ Explain Round Robin CPU scheduling with a Gantt chart concept.
40s OS
Round Robin (RR) is a preemptive CPU scheduling algorithm designed for time-sharing systems. * **How it works:** Each process is assigned a small, fixed unit of CPU time called a **Time Quantum** (typically 10-100ms). Processes are placed in a FIFO queue. The CPU scheduler picks the first process, runs it for up to one quantum, then preempts it and places it at the end of the queue. * **Gantt Chart Example:** If P1 (burst=8), P2 (burst=4), P3 (burst=3) arrive at time 0, and time quantum = 3: * Gantt chart: `[P1: 0-3] -> [P2: 3-6] -> [P3: 6-9] -> [P1: 9-12] -> [P2: 12-13] -> [P1: 13-15]` * **Tradeoff:** If time quantum is too large, RR behaves like FCFS. If too small, context switching overhead dominates CPU time.
Key points: Preemptive schedulingTime QuantumFIFO ready queueQuantum sizing tradeoff
⚠️ Do NOT say: Forgetting to mention the context-switching overhead when the time quantum is too small
✓ What is the difference between a Mutex and a Semaphore?

✓ Explain the Producer-Consumer problem and its semaphore-based solution.
50s OS
The Producer-Consumer problem is a classic synchronization problem where a producer thread generates data and writes it to a fixed-size buffer, while a consumer thread reads and processes it concurrently. **Issues:** 1. Producer must not add data to a full buffer (Buffer Overflow). 2. Consumer must not read from an empty buffer (Buffer Underflow). 3. Critical sections must be mutually exclusive. **Solution using Semaphores:** * Use three semaphores: 1. `mutex = 1` (Binary semaphore for mutual exclusion on buffer writes). 2. `empty = N` (Counting semaphore tracking empty slots). 3. `full = 0` (Counting semaphore tracking filled slots). *Producer Logic:* ```python wait(empty) wait(mutex) # Add item to buffer signal(mutex) signal(full) ``` *Consumer Logic:* ```python wait(full) wait(mutex) # Remove item from buffer signal(mutex) signal(empty) ```
Key points: Fixed-size bufferBuffer overflow/underflowThree semaphoresMutex lockEmpty/Full signaling
⚠️ Do NOT say: Omitting the empty and full counting semaphores, or getting the wait/signal order mixed up
✓ What is context switching and what is its cost?
40s OS
Context switching is the CPU mechanism of saving the state of the active process/thread (PCB/TCB registers, program counter) and loading the state of a ready process/thread to begin execution. **The Costs involved:** 1. **CPU Overhead:** Saving and restoring registers takes clock cycles. 2. **TLB Flush:** Changing page tables in process switching invalidates the Translation Lookaside Buffer (TLB). Subsequent memory lookups are slow (must query RAM page tables). 3. **CPU Cache Misses:** The active L1/L2 caches are cold for the new process, slowing down execution initially.
Key points: PCB/TCB state saveTLB invalidationCache coldness overheadCPU cycles cost
⚠️ Do NOT say: Describing context switching as instant or neglecting the TLB flushing cost

Key points: NewReadyRunningBlocked/WaitingTerminatedLifecycle transitions

✓ Explain the Banker's algorithm with an illustrative example concept.
45s OS
The Banker's Algorithm is a resource allocation and deadlock avoidance algorithm that tests for safety by simulating resource allocation requests and verifying if the system remains in a safe state. **Core Data Structures:** * **Available:** Array representing available resources of each type. * **Max:** Matrix defining maximum demand of each process. * **Allocation:** Matrix defining resources currently allocated to each process. * **Need:** Matrix calculating remaining resources needed (`Need = Max - Allocation`). **Safe State Concept:** * A state is safe if there exists a **Safe Sequence** of processes such that the remaining needs of all processes can be satisfied. * If a request leaves the system with no sequence to satisfy all needs sequentially, the request is denied and the process blocks.
