227### "What is RAG and how does it work?"
228
229"RAG stands for Retrieval Augmented Generation. The core idea is: instead of relying on what the LLM memorized during training, you retrieve relevant context at query time and inject it into the prompt.
230
231The pipeline has three parts. First, indexing — you take your documents, chunk them, convert each chunk into an embedding using a model like text-embedding-3-small, and store those vectors in a vector database like pgvector or Pinecone. Second, retrieval — when the user asks a question, you convert that query into an embedding too, then do a cosine similarity search to find the most relevant chunks. Third, generation — you pass those retrieved chunks as context into the LLM prompt and let it generate a grounded answer.
232
233I've implemented this in both Auren and Mnemo. In Mnemo specifically, I added a decay scoring layer on top of cosine similarity so more recent memories are weighted higher, and a contradiction detection step that flags if a new memory conflicts with an existing one."
234

237### "What is a vector database and why do we use it?"
238
239"A regular database stores structured data and lets you filter by exact values. A vector database stores high-dimensional numerical vectors — embeddings — and lets you search by semantic similarity. The key operation is approximate nearest neighbor search.
240
241When you embed text, semantically similar text ends up close together in vector space. So 'how to cook pasta' and 'pasta recipe' will be near each other even though they share no exact words. That's something a SQL LIKE query cannot do.
242
243I use pgvector — it's a PostgreSQL extension that adds vector types and operators. The index type I use is HNSW — Hierarchical Navigable Small World — which gives you sub-linear search time even on large datasets."

247### "Explain the Transformer architecture"
248
249"The Transformer came out of the 'Attention Is All You Need' paper from Google in 2017. The key insight was: you don't need RNNs to process sequences. Attention alone is enough.
250
251The architecture has an encoder and decoder. The encoder takes input tokens, converts them to embeddings, adds positional encoding so the model knows token order, and then passes them through multi-head self-attention layers followed by feedforward layers. Self-attention lets every token attend to every other token — so it captures long-range dependencies that RNNs struggled with.
252
253The decoder does the same but adds one extra step — cross-attention, which lets each output token attend to the encoder's representations.
254
255BERT uses only the encoder, trained with masked language modeling. GPT uses only the decoder, trained to predict the next token. For chat models you mostly deal with GPT-style architectures."

259### "What is the difference between fine-tuning, prompt engineering, and RAG?"
260
261"These are three different levers to make an LLM do what you want.
262
263Prompt engineering is the lightest — you craft your input carefully, maybe add examples in the prompt, structure it as chain-of-thought or few-shot. No training required. Best for general-purpose tasks where the model already has the knowledge.
264
265RAG gives the model access to external knowledge at inference time. You don't change the model weights — you just inject relevant context into the prompt. Best when the model lacks specific domain knowledge or your data changes frequently.
266
267Fine-tuning actually updates the model's weights using your dataset. The model learns your style, your format, your specific vocabulary. Best when you need consistent output format, domain-specific behavior, or you have lots of labeled examples. It's expensive and requires GPU access.
268
269For most startup use cases, RAG is the right choice. Fine-tuning makes sense when you've already exhausted prompt engineering and RAG."

273### "What is OOP? Explain with Python examples"
274
275"OOP stands for Object Oriented Programming. It's a paradigm where you model real-world entities as objects that have attributes (data) and methods (behavior).
276
277Four pillars.
278
279Encapsulation — wrapping data and the methods that operate on it inside a class, and controlling access. In Python, you prefix with underscore for convention-based privacy. Not enforced at language level like Java but respected by convention.
280
281Inheritance — a class can inherit from another class and reuse its attributes and methods. Python supports multiple inheritance. MRO (Method Resolution Order) using C3 linearization determines which parent's method gets called.
282
283Polymorphism — same method name, different behavior. In Python this is mostly duck typing — if it has the method, it works. You don't need explicit interfaces.
284
285Abstraction — hiding implementation details and exposing only what's necessary. Python's ABC module lets you define abstract classes with abstract methods that subclasses must implement."

289### "What are ACID properties in DBMS?"
290
291"ACID is a set of properties that guarantee database transactions are reliable.
292
293Atomicity — a transaction is all or nothing. If you're transferring money between accounts, either both the debit and credit happen, or neither does.
294
295Consistency — the database must go from one valid state to another. No rules are violated after a transaction commits.
296
297Isolation — concurrent transactions don't interfere with each other. Each transaction sees a consistent snapshot.
298
299Durability — once a transaction commits, it stays committed even if the system crashes. It's written to disk, not just in memory.
300
301SQL databases are ACID-compliant by default. NoSQL databases often trade some of these (especially isolation and consistency) for scalability and speed."
302

305### "Explain normalisation"
306
307"Normalization is the process of organizing a database to reduce redundancy and avoid anomalies — insert, update, and delete anomalies.
308
309First Normal Form (1NF): every column has atomic values. No arrays or repeating groups inside a cell.
310
311Second Normal Form (2NF): 1NF plus every non-key column must depend on the entire primary key, not just part of it. This matters for composite primary keys.
312
313Third Normal Form (3NF): 2NF plus no transitive dependency. Non-key columns should not depend on other non-key columns.
314
315BCNF (Boyce-Codd Normal Form): stricter version of 3NF. Every determinant must be a candidate key.
316
317In practice, most production databases normalize to 3NF and sometimes denormalize deliberately for read performance."
318

321### "What is deadlock? When does it occur?"
322
323"A deadlock is a state where two or more processes are stuck, each waiting for a resource held by the other, so none of them can proceed.
324
325Four conditions must all be true simultaneously for a deadlock to occur. Mutual exclusion — at least one resource is non-shareable. Hold and wait — a process holding one resource is waiting for another. No preemption — resources can't be forcibly taken from a process. Circular wait — there's a cycle of processes each waiting on the next.
326
327To prevent deadlock you break one of these conditions. Common strategies: impose a fixed resource ordering to break circular wait, use resource allocation graphs, or use the Banker's Algorithm which checks before granting a resource whether the system will remain in a safe state."
328

331### "What is the difference between process and thread?"
332
333"A process is an independent program in execution. It has its own memory space — stack, heap, code, data. Processes are isolated from each other. If one crashes, others are unaffected.
334
335A thread is the smallest unit of execution within a process. Threads in the same process share memory — they share the heap and the code segment, but each has its own stack and registers. That shared memory is what makes threads faster to communicate but also harder to program safely — you need synchronization to avoid race conditions.
336
337Context switching between threads is cheaper than between processes because threads share the same memory space."

341### "Write a query: find the second highest salary from an Employee table"
342
343"There are two clean ways.
344
345First approach using subquery:
346SELECT MAX(salary) FROM Employee WHERE salary < (SELECT MAX(salary) FROM Employee);
347
348Second approach using DENSE_RANK or LIMIT:
349SELECT salary FROM Employee ORDER BY salary DESC LIMIT 1 OFFSET 1;
350
351The subquery approach works across all SQL dialects. The LIMIT/OFFSET approach works in MySQL and PostgreSQL. If duplicates matter, use DENSE_RANK in a window function."

156"What is RAG? Have you implemented it?"
157
158"Yeah, I've built it twice so let me explain from how I think about it rather than the textbook definition.
159
160The core problem RAG solves is this: an LLM knows what it learned during training but it has no idea what's in your company's internal docs, your emails, your codebase. You can't fine-tune it every time something changes. RAG says — okay, at the time the user asks something, go retrieve the relevant information and hand it to the model as context. The model then answers based on what it just received, not just what it memorised.
161
162The pipeline has three parts. Indexing: you take your documents, break them into chunks, convert each chunk to a vector embedding using an embedding model, store those in a vector database. Retrieval: when someone asks a question, you embed that question too, then find chunks that are semantically closest to it using cosine similarity. Generation: you inject those chunks into the prompt and let the LLM generate a grounded answer.
163
164In Mnemo — my memory API — I added two layers on top of vanilla RAG. One is decay scoring: recent memories get a higher weight even if an older memory has slightly better cosine similarity. Because in most real use cases, what you said last week matters more than what you said six months ago. The second is contradiction detection: if a new memory conflicts with an existing one, the system flags it instead of silently storing both and confusing the model later.
165
166So yes, I've implemented it, and I've thought beyond the basic version."

170"Explain a Transformer to me"
171
172"So before transformers we had RNNs, and the fundamental problem with RNNs was they processed sequences token by token, in order. Which means if you're reading a sentence and the important context is 50 words back, by the time you get to it the earlier information has basically faded out. The network forgets.
173
174The transformer's insight was: what if every word can look at every other word at the same time? That's what attention is. Each token asks 'how relevant is every other token to me right now?' and you get a weighted sum. You're not reading left to right — you're looking at the whole thing at once.
175
176Multi-head attention runs this process multiple times in parallel — each head learns to focus on different kinds of relationships. One head might focus on syntactic structure, another on semantic meaning. You run 8 or 12 heads, concatenate the outputs, and you've captured richer relationships than any single attention head could.
177
178Then there's positional encoding because the attention mechanism itself has no concept of order. You add a position vector to each token embedding to tell the model where in the sequence each token sits.
179
180The encoder builds a contextual representation of the input. The decoder generates the output, and it uses cross-attention to look at the encoder's representations while it's generating each output token.
181
182BERT uses just the encoder — it was trained on masked language modelling, so it learned to predict missing words. GPT uses just the decoder, trained to predict the next token. For most chat applications you're working with GPT-style decoder-only architectures."

186"What's the difference between fine-tuning, RAG, and prompt engineering?"
187
188"They're three different levers and they're not interchangeable — each one solves a different problem.
189
190Prompt engineering is the cheapest and fastest. You don't touch the model at all. You just craft your input more carefully — add examples, structure your request, break complex tasks into steps. Zero-shot just tells the model what to do. Few-shot gives it 2–3 examples of the format you want. Chain-of-thought asks it to reason step by step before answering. Works well when the model already has the knowledge and you just need to steer the output.
191
192RAG is for when the model doesn't have the knowledge — or the knowledge changes frequently. You're not changing the model, you're giving it fresh information at the time of each query. I use this in Auren for email context and in Mnemo for persistent memory. The limitation is that retrieval quality matters a lot — if you retrieve the wrong chunks, the model will still sound confident while being wrong.
193
194Fine-tuning actually changes the model's weights using your labelled dataset. This is for when you need the model to write in a specific tone consistently, follow a very specific output format every time, or deeply understand a domain-specific vocabulary. It's expensive, you need GPU access, and you need a good dataset. For most startup use cases RAG is the right call. Fine-tuning makes sense after you've exhausted the other two and you have clear evidence they're not sufficient."

198"Explain OOPs. Can you give me examples?"
199
200"Sure. OOPs is basically a way to organise code around objects instead of just functions running in sequence. The four pillars —
201
202Encapsulation is about bundling data and the functions that operate on it inside one unit, and deciding what the outside world gets to see. In Python you signal private stuff with an underscore prefix — it's convention, not hard enforcement. The benefit is you can change how something works internally without breaking everything that uses it.
203
204Inheritance is when one class gets the attributes and methods of another. Saves you from rewriting the same logic. Python supports multiple inheritance — one class can inherit from multiple parents — and uses something called MRO, Method Resolution Order, to figure out which parent's version of a method gets called when there's a conflict. It follows C3 linearisation.
205
206Polymorphism is the same method name doing different things depending on what object you're calling it on. In Python this is largely duck typing — if the object has the method, it works. You don't need explicit interfaces like in Java.
207
208Abstraction is hiding the implementation and just exposing what the user of your class needs. Python has an ABC module — Abstract Base Classes — where you can define methods that every subclass must implement. If a subclass doesn't implement a required abstract method, you get an error when you try to instantiate it."

212"What are ACID properties?"
213
214"ACID is what makes database transactions reliable, especially when multiple things are happening at the same time or something crashes mid-operation.
215
216Atomicity is all or nothing. Classic example: you're transferring money between two bank accounts. The debit from account A and the credit to account B need to happen together. If the system crashes after the debit but before the credit, atomicity rolls the whole thing back. You don't end up with money disappearing.
217
218Consistency means the database goes from one valid state to another. Whatever rules and constraints exist — foreign keys, unique constraints, check constraints — they all hold after every transaction.
219
220Isolation is about concurrent transactions not interfering with each other. If two transactions are running at the same time, each one should see a consistent snapshot, not half-finished work from the other. Different isolation levels give you different trade-offs between correctness and performance.
221
222Durability means once a transaction says it committed, it's committed. Even if the machine crashes immediately after, the data is persisted to disk. It's not just in memory."
223

226"What is normalisation?"
227
228"Normalisation is about organising your database so you're not storing the same information in multiple places, which causes problems when you try to update it.
229
230First Normal Form just says every cell in your table should hold one value, not a list. If you're storing comma-separated values in a single column, that's a 1NF violation.
231
232Second Normal Form builds on that by saying if you have a composite primary key, every non-key column needs to depend on the whole key, not just one part of it. Partial dependency is what it eliminates.
233
234Third Normal Form eliminates transitive dependency — which means non-key columns shouldn't depend on other non-key columns. If knowing the zip code tells you the city, and the city isn't part of the primary key, that's a transitive dependency. You'd split that into its own table.
235
236BCNF is slightly stricter than 3NF. It says every determinant — any column that uniquely determines another column — must be a candidate key.
237
238In real production systems you normalise to 3NF and then sometimes deliberately denormalise specific tables where you need faster reads. It's a trade-off."

242"What is a deadlock?"
243
244"A deadlock is when two or more processes are stuck, each one waiting for a resource that another one is holding, and none of them can proceed because they're all waiting on each other.
245
246Four conditions have to all be true at the same time for a deadlock to happen. First, mutual exclusion — at least one resource can only be used by one process at a time. Second, hold and wait — a process is holding a resource and also waiting to get more. Third, no preemption — you can't forcibly take a resource away from a process, it has to release it voluntarily. Fourth, circular wait — you have a cycle where process A is waiting for process B which is waiting for process C which is waiting for process A.
247
248To prevent it you break one of those conditions. The most common approach is imposing a fixed ordering on resources — if all processes always request resources in the same order, you can never get a circular wait. The Banker's Algorithm is a more sophisticated approach where before granting a resource, the system checks whether the resulting state is still safe — meaning there's at least one sequence in which all processes can complete."

252"What's the difference between a process and a thread?"
253
254"A process is an independent running program with its own memory space — its own heap, stack, code, data. Processes are isolated. If one crashes, it doesn't take down others because they're not sharing memory.
255
256A thread lives inside a process. Multiple threads within the same process share the heap and the code, but each thread has its own stack and its own registers. They're lighter than processes because you're not duplicating memory. Switching between threads is also cheaper than switching between processes.
257
258The trade-off is that because threads share memory, you can get race conditions — two threads reading and writing the same data at the same time in an undefined order. That's why you need synchronization like mutexes or semaphores."
259
