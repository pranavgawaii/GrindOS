What is Database Indexing?

Database Indexing is a technique used to optimize the performance of a database by reducing the time required to retrieve specific records. An index acts like a table of contents in a book, pointing to the location of data without scanning the entire database.

Why Is Indexing Important?

  • Faster Query Execution: Speeds up SELECT operations.
  • Efficient Data Retrieval: Reduces search time.
  • Improved Performance: Minimizes disk I/O operations.

Key Concept

Indexes are created on columns that are frequently searched, sorted, or filtered in queries. They help the database management system (DBMS) locate rows efficiently.

Real-Life Relation:

Consider a library catalog. Instead of scanning every shelf to find a specific book, you search the catalog, which quickly points you to the correct location. Similarly, database indexes allow fast lookup of records without scanning the entire table.

Example Table

Table: Bookings

Passenger BookingID Date
Raj 201 2024-10-01
Gauri 202 2024-10-01
Liza 203 2024-10-02

Index Created on Passenger Column:

Passenger Pointer to Disk Location
Raj 149
Gauri 2459
Liza 3368

With this index, a query like SELECT * FROM bookings WHERE Passenger = 'Liza' retrieves data directly using the pointer 3368, avoiding a full table scan.


Structure of an Index in a Database

An index is usually implemented using data structures such as:

  • B-Trees: Balanced tree structures for indexing sorted data.
  • Hash Maps: Fast lookups using hash functions for equality searches.

Example:

Assume a table students with the following data:

ID Name Age
101 Alice 20
102 Bob 22
103 Charlie 21

If an index is created on the Name column, the database organizes it like this:

Index (Name): Alice -> Row 1  
                Bob -> Row 2  
             Charlie -> Row 3

Using this index, a query like SELECT * FROM students WHERE Name = 'Bob' is executed faster as it directly refers to Row 2.


Attributes of Indexing

  • Unique Index: Ensures that values in the indexed column are unique.

    Example: Index on primary key columns.

  • Non-Unique Index: Allows duplicate values in the indexed column.

    Example: Index on a column like Age.

  • Clustered Index: Physically rearranges the table rows to match the index order.

    Example: Primary key indexes in most DBMS.

  • Non-Clustered Index: Creates a separate structure for the index, without changing the physical table order.

    Example: Indexes on non-primary key columns.

Types of File Organisation

File organization determines how database records are stored on disk, affecting both storage efficiency and retrieval speed.

1. Sequential File Organisation

Records are stored in sequential order, sorted by the primary key.

  • Advantages:
    • Efficient for range queries.
    • Simple to implement.
  • Disadvantages:
    • Slow for insertions or deletions as records must be reordered.

Example: A file storing employee records sequentially by ID:

[101] -> [102] -> [103] -> [104]

To add a new employee with ID 102.5, all subsequent records must shift.

2. Hash File Organisation

Records are stored based on a hash function applied to the key, mapping it to a specific location.

  • Advantages:
    • Very fast for equality searches.
    • Efficient for lookups.
  • Disadvantages:
    • Inefficient for range queries.
    • Collisions can degrade performance.

Example: Using a hash function h(ID) = ID % 10:

[101] -> Bucket 1  
[102] -> Bucket 2  
[103] -> Bucket 3

Comparison of Sequential and Hash File Organisation

Feature Sequential Hash
Storage Order Sorted Based on hash function
Search Type Efficient for range queries Efficient for equality queries
Insert/Delete Speed Slower due to reordering Faster due to direct mapping
