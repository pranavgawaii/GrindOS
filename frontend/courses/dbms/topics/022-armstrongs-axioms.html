Armstrong’s Axioms

Armstrong’s Axioms are a set of inference rules used to derive all functional dependencies in a database. These rules help maintain data integrity and normalization in relational database design. They are sound (produce only correct results) and complete (derive all valid functional dependencies).

Real-Life Relation

In a database, if A → B and B → C are known, Armstrong’s axioms allow us to infer that A → C. This ensures the database schema is consistent and logically sound.

Reflexivity

If a set of attributes includes another subset, the larger set functionally determines the smaller subset.

Rule

If Y ⊆ X, then X → Y.

Example

If A and B are attributes, then AB → A.

Real-Life Relation

In a student database, if a table contains columns StudentID and Name, then StudentID, Name → Name.

Augmentation

If X → Y, then adding any set of attributes Z to both sides preserves the dependency.

Rule

If X → Y, then XZ → YZ.

Example

If A → B, then AC → BC.

Real-Life Relation

If knowing a customer’s CustomerID determines their Name, then knowing CustomerID and their Phone will also determine their Name and Phone.

Transitivity

Definition and Explanation

If X → Y and Y → Z, then X → Z.

Rule

If X → Y and Y → Z, then X → Z.

Example

If A → B and B → C, then A → C.

Real-Life Relation

In a library database:

  • ISBN → BookTitle
  • BookTitle → Author
  • Therefore, ISBN → Author.

Self-Determination

Rule

If X → X.

Example

A → A.

Real-Life Relation

In any database, a column like StudentID determines itself (StudentID → StudentID).

Decomposition

Rule

If X → YZ, then X → Y and X → Z.

Example

If A → BC, then A → B and A → C.

Real-Life Relation

In a sales database:

  • OrderID → (CustomerID, ProductID)
  • This implies OrderID → CustomerID and OrderID → ProductID.

Union

Rule

If X → Y and X → Z, then X → YZ.

Example

If A → B and A → C, then A → BC.

Real-Life Relation

In a database:

  • EmployeeID → Name
  • EmployeeID → Department
  • Therefore, EmployeeID → (Name, Department).

Derived Rules- Composition

Rule

If X → Y and Z → W, then XZ → YW.

Example

If A → B and C → D, then AC → BD.

Real-Life Relation

In a product database:

  • ProductID → ProductName
  • SupplierID → SupplierName
  • Therefore, ProductID, SupplierID → ProductName, SupplierName.
