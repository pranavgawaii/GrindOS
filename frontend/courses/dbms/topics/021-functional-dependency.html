What is a Functional Dependency?

Functional Dependency (FD) is a relationship between attributes in a relational database. It states that the value of one attribute (or a group of attributes) uniquely determines the value of another attribute. In other words, if we know the value of one attribute, we can predict the value of the dependent attribute(s) based on this relationship.

Notation

If attribute Y is functionally dependent on attribute X, it is written as: X → Y. This means that for any two rows in the table with the same value of X, the value of Y will also be the same.

Example

In a Student table, if Student_ID uniquely determines Name, then Student_ID → Name is a functional dependency.

Why is Functional Dependency Important?

  • Foundation of Normalization: FDs help identify redundancies and anomalies, enabling database normalization to reduce duplication and maintain consistency.
  • Ensures Data Integrity: By establishing dependencies, FDs ensure that data remains accurate and consistent throughout the database.
  • Database Design Optimization: FDs provide insight into which attributes should be kept together, helping optimize database structure by minimizing storage and retrieval costs.

In short, functional dependencies play a key role in database design by influencing table decomposition, normalization, and integrity.

Trivial vs. Non-Trivial Functional Dependency

1. Trivial Functional Dependency

A functional dependency is trivial if the dependent attribute is a subset of the determinant attribute(s). In simple terms, X → Y is trivial if Y is part of X.

Example

If X = {Student_ID, Name}, then:

{Student_ID, Name} → Student_ID is a trivial dependency since Student_ID is part of {Student_ID, Name}.

2. Non-Trivial Functional Dependency

A functional dependency is non-trivial if the dependent attribute is not a subset of the determinant attribute(s). Non-trivial dependencies are important because they reveal meaningful relationships between different attributes in a table.

Example

In a Course table, if Course_ID uniquely determines Instructor, then Course_ID → Instructor is a non-trivial dependency because Instructor is not part of Course_ID.

Why This Matters: Trivial dependencies often don’t contribute to database design decisions, whereas non-trivial dependencies reveal meaningful attribute relationships that guide normalization and decomposition.

Inference Rules

Inference Rules (also called Armstrong’s Axioms) are a set of rules used to derive all possible functional dependencies from a given set of FDs. These rules help determine the logical implications of FDs and are essential in determining whether a database is in a normalized form.

Primary Inference Rules

  • Reflexivity: If Y is a subset of X, then X → Y.
    Example: {Student_ID, Name} → Student_ID since Student_ID is part of {Student_ID, Name}.
  • Augmentation: If X → Y, then XZ → YZ for any attribute Z.
    Example: If Student_ID → Name, then {Student_ID, Major} → {Name, Major}.
  • Transitivity: If X → Y and Y → Z, then X → Z.
    Example: If Course_ID → Instructor and Instructor → Office, then Course_ID → Office.

Secondary Inference Rules

  • Union: If X → Y and X → Z, then X → YZ.
    Example: If Course_ID → Title and Course_ID → Credits, then Course_ID → {Title, Credits}.
  • Decomposition: If X → YZ, then X → Y and X → Z.
    Example: If Student_ID → {Name, Major}, then Student_ID → Name and Student_ID → Major.
  • Pseudo Transitivity: If X → Y and YW → Z, then XW → Z.
    Example: If Student_ID → Major and Major, GPA → Honor_Status, then Student_ID, GPA → Honor_Status.

Closure of Functional Dependency

The closure of a set of functional dependencies is the complete set of all FDs that can be derived from a given set of FDs using the inference rules. The closure helps identify all possible dependencies among attributes, enabling better decisions regarding database decomposition and normalization.

How to Find the Closure of an Attribute Set (X+):

  1. Start with X+: Begin with the given attribute set X.
  2. Apply FDs: Use the given FDs and inference rules to add attributes to X+ if they are determined by X.
  3. Repeat: Continue until no more attributes can be added to X+.

Example

Given FDs: A → B, B → C, and C → D, to find the closure of {A}, we:

  • Start with {A}.
  • Since A → B, add B to {A}, giving {A, B}.
  • Since B → C, add C to {A, B}, resulting in {A, B, C}.
  • Since C → D, add D to {A, B, C}, giving {A, B, C, D}.

Therefore, the closure of {A} is {A, B, C, D}.

Example

Consider a Student Enrollment System with attributes: Student_ID, Course_ID, Instructor, and Department. Let’s say we have the following FDs:

  • Student_ID → Course_ID (a student is enrolled in a specific course).
  • Course_ID → Instructor (each course has a specific instructor).
  • Instructor → Department (each instructor belongs to a department).

Steps to Normalize Using Functional Dependencies:

  1. Identify Functional Dependencies: Given dependencies reveal that Student_ID determines Course_ID, Course_ID determines Instructor, and Instructor determines Department.
  2. Determine Closure: Using FDs, find closures to identify unique determinants and reduce redundancies.
  3. Decompose as Needed: Using these dependencies, the table can be decomposed into tables based on normalization requirements.
