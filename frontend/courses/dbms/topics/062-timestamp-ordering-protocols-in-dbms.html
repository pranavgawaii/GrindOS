Timestamp Ordering Protocols in DBMS

The Time Stamping Model is a concurrency control mechanism that assigns a unique timestamp to each transaction when it starts. These timestamps establish the order of transactions, ensuring that older transactions have priority over newer ones. By ordering transactions based on their timestamps, the database can prevent conflicts and avoid the need for complex locking mechanisms.

Key Concepts

  • Timestamp: A unique identifier (often a timestamp or sequence number) assigned to each transaction.
  • Ordering: Transactions are executed in the order of their timestamps.
  • Conflict Prevention: Transactions that violate timestamp order are either delayed or aborted, ensuring consistency.

Wait-Die Scheme

The Wait-Die Scheme is a timestamp-based deadlock prevention protocol. In this scheme:

  • Older transactions can wait for younger transactions.
  • Younger transactions must abort (or "die") if they attempt to access a resource held by an older transaction.

Example

Transaction A (older) has a timestamp of 100. Transaction B (younger) has a timestamp of 200:

  • If Transaction B attempts to access a resource held by Transaction A: Transaction B will "die" (abort and restart) because it is younger.
  • If Transaction A attempts to access a resource held by Transaction B: Transaction A will wait because it has an older timestamp.

Wound-Wait Scheme

The Wound-Wait Scheme is another timestamp-based deadlock prevention protocol with rules opposite to the Wait-Die Scheme:

  • Older transactions "wound" (forcefully abort) younger transactions holding the desired resource.
  • Younger transactions wait if they attempt to access a resource held by an older transaction.

Example

Transaction A (older) has a timestamp of 100. Transaction B (younger) has a timestamp of 200:

  • If Transaction A wants a resource held by Transaction B: Transaction A will "wound" (abort) Transaction B, and Transaction B restarts.
  • If Transaction B wants a resource held by Transaction A: Transaction B will wait, allowing Transaction A to proceed.

Types of Timestamp Values

Timestamp Type Description
Read Timestamp (RTS) The most recent timestamp of a transaction that successfully read the data item.
Write Timestamp (WTS) The most recent timestamp of a transaction that successfully modified the data item.
Commit Timestamp (CTS) The timestamp at which the transaction commits, ensuring no future transactions can access earlier states.

Timestamp Ordering Protocols

Timestamp Ordering Protocols use the assigned timestamps to enforce a specific order of transaction execution, ensuring a conflict-free sequence. When a transaction attempts to read or write a data item, the protocol checks the transaction’s timestamp against the data item’s timestamps to determine if the action is allowed.

  • Read Rule: A transaction can only read a data item if its timestamp is greater than the data item’s latest write timestamp (WTS). If the transaction’s timestamp is less than the WTS, the transaction is aborted.
  • Write Rule: A transaction can only write a data item if its timestamp is greater than the data item’s latest read (RTS) and write timestamps (WTS). If the transaction’s timestamp is less than either the RTS or WTS, the transaction is aborted.

Conclusion

Timestamp-based concurrency control mechanisms like Wait-Die, Wound-Wait, and Thomas’ Rules ensure transaction consistency and reduce conflicts in databases. By enforcing strict ordering and leveraging timestamp values, these protocols enable efficient and conflict-free transaction execution.
