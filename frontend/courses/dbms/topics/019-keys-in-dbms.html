Types of Keys in Database

Keys are attributes or combinations of attributes that uniquely identify rows within a table in a relational database. They are critical in defining relationships between tables, ensuring data integrity, and preventing duplicate records. Different types of keys serve various roles within a database, from identifying records to establishing connections between tables.

1. Primary Key

A Primary Key is a unique identifier for each record in a table. It ensures that no two rows have the same primary key value, thus maintaining data integrity. Each table can have only one primary key, and the values in the primary key column(s) cannot be null.

  • Characteristics: Uniquely identifies each record, enforces uniqueness and non-null constraints, and each table has only one primary key.

Example: In a Student table, Student_ID could serve as a primary key:

Student_ID Name Age
101 Alice 22
102 Bob 23

2. Candidate Key

A Candidate Key is an attribute, or a set of attributes, that can uniquely identify a row in a table. There can be multiple candidate keys in a table, but only one can be chosen as the primary key. Candidate keys must be unique and should not contain null values.

  • Characteristics: Can uniquely identify each record, multiple candidate keys can exist in a single table, and one of the candidate keys is chosen as the primary key.

Example: In an Employee table with attributes Employee_ID and Social_Security_Number (SSN), both Employee_ID and SSN can uniquely identify each employee, making them candidate keys.

3. Alternate Key

An Alternate Key is a candidate key that is not selected as the primary key. It is an alternate way of uniquely identifying a record within a table, and it maintains the uniqueness requirement.

  • Characteristics: Serves as a unique identifier like the primary key, and is one of the candidate keys that wasn’t chosen as the primary key.

Example: In the Employee table, if Employee_ID is chosen as the primary key, then SSN becomes an alternate key.

4. Composite Key

A Composite Key is a combination of two or more attributes that together uniquely identify a row. Composite keys are used when a single attribute cannot uniquely identify records.

  • Characteristics: Consists of two or more attributes, used when no single attribute can serve as a unique identifier, and each attribute in a composite key may not be unique on its own.

Example: In an Order table, both Order_ID and Product_ID together uniquely identify each row:

Order_ID Product_ID Quantity
5001 101 3
5001 102 1

5. Super Key

A Super Key is any combination of attributes that can uniquely identify a row in a table. Super keys include candidate keys as well as other sets of attributes that uniquely identify records.

  • Characteristics: Contains at least one attribute or a combination that uniquely identifies records, and can include extra attributes beyond what is required to ensure uniqueness.

Example: In a Student table, both {Student_ID} and {Student_ID, Name} are super keys, as both uniquely identify a student, but only {Student_ID} is a candidate key (minimal super key).

6. Foreign Key

A Foreign Key is a column or set of columns in one table that establishes a link to the primary key of another table. Foreign keys are used to create relationships between tables and maintain referential integrity.

  • Characteristics: Links two tables by referring to the primary key of another table, and enforces referential integrity.

Example: In an Enrollment table, Student_ID can be a foreign key referencing the Student table’s primary key Student_ID to establish a relationship between the tables:

Enrollment_ID Student_ID Course_ID
1 101 CS101
2 102 MA202

7. Unique Key

A Unique Key is a constraint that ensures all values in a column are unique across the table. Unique keys can have null values (typically one null), unlike primary keys, which cannot be null.

  • Characteristics: Enforces uniqueness across values in a column or set of columns, and can allow null values (usually one null per unique key).

Example: In a User table, Email can be a unique key, as each user’s email must be distinct:

User_ID Name Email
1 Alice alice@example.com
2 Bob bob@example.com

Key Type Description Example
Primary Key Unique identifier for each row; does not allow nulls. Student_ID in Student table
Candidate Key Attribute(s) that can uniquely identify rows; one is chosen as the primary key. Employee_ID and SSN in Employee table
Alternate Key Candidate key not chosen as primary key. SSN if Employee_ID is the primary key
Composite Key Combines two or more attributes to uniquely identify rows. {Order_ID, Product_ID} in Order table
Super Key Any set of attributes that uniquely identifies rows, possibly with extra attributes. {Student_ID}, {Student_ID, Name}
