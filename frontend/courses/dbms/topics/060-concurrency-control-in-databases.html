What is Concurrency Control?

Concurrency Control is the process of managing simultaneous transactions in a multi-user database system to prevent conflicts and ensure data consistency. In a database, multiple users may try to access or modify the same data at the same time, which can lead to issues like lost updates, dirty reads, and inconsistent data. Concurrency control ensures that transactions execute in a manner that maintains the database's integrity.

Example:

In an Online Banking System, if two users simultaneously attempt to withdraw money from the same account, concurrency control ensures that only one transaction completes successfully, avoiding potential overdrafts or data inconsistencies.


Why Do We Need Concurrency Control?

Concurrency control is essential for maintaining the ACID (Atomicity, Consistency, Isolation, Durability) properties of a transaction. Without effective concurrency control, the following issues may arise:

  • Lost Updates: Occurs when two transactions update the same data simultaneously, with one update overwriting the other.
  • Dirty Reads: Occurs when a transaction reads data that has been modified by another transaction that has not yet committed, leading to inconsistent views of the data.
  • Uncommitted Dependency (Cascading Rollback): If a transaction depends on uncommitted changes by another transaction, it may also fail if the initial transaction is rolled back.
  • Inconsistent Analysis: Occurs when a transaction reads data during an update by another transaction, resulting in inconsistencies.

Example of Dirty Read:

Consider a reIlon inventory system where one transaction updates product prices and another reads those prices to calculate sales. Without concurrency control, the sales calculation may use partially updated prices, leading to incorrect results.


Concurrency Control Techniques

There are several primary techniques used for concurrency control:

  • Lock-Based Protocols: Transactions acquire locks on data items to control access and prevent conflicts.
  • Timestamp-Based Protocols: Transactions are assigned timestamps to ensure they execute in a specific order.
  • Optimistic Concurrency Control: Assumes minimal conflict and validates transactions before committing, reducing locking overhead.

Lock-Based Protocols

Lock-Based Protocols use locks to control access to data. A lock restricts access to a data item, ensuring that only one transaction can modify or read it at a time.

Types of Locks:

  • Shared Lock (S-Lock): Allows a transaction to read a data item but not modify it. Multiple transactions can hold shared locks on the same item simultaneously.
  • Exclusive Lock (X-Lock): Allows a transaction to both read and modify a data item. Only one transaction can hold an exclusive lock on a data item at a time.

Two-Phase Locking (2PL):

Two-Phase Locking (2PL) is a commonly used lock-based protocol that divides the locking process into two phases:

  • Growing Phase: The transaction acquires all required locks but cannot release any.
  • Shrinking Phase: The transaction releases locks and cannot acquire any new locks.

2PL ensures serializability but can lead to deadlocks, where two or more transactions wait indefinitely for each other to release locks.


Example:

In an Online Shopping Cart system:

  • A shared lock allows multiple users to view an item’s details.
  • An exclusive lock allows only one user to modify the item’s quantity or add it to the cart.

Timestamp-Based Protocols

Timestamp-Based Protocols assign each transaction a unique timestamp when it starts. This timestamp determines the order of transactions, ensuring they execute in a way that prevents conflicts.

How It Works:

  • Timestamp Ordering: Transactions are ordered by their timestamps, ensuring older transactions execute before newer ones.
  • Read and Write Checks: Before a transaction reads or writes a data item, the system checks timestamps to ensure the transaction is allowed to proceed without conflict.

Timestamp-based protocols avoid deadlocks, as transactions do not wait for each other, but they may lead to cascading rollbacks if a transaction needs to be aborted.

Example:

In a Library System, if two users check out a book simultaneously, the transaction with the earlier timestamp gets priority. This ensures that the system follows a consistent order of actions.

Optimistic Concurrency Control

Optimistic Concurrency Control (OCC) assumes that conflicts are rare and minimizes locking by allowing transactions to execute without initial checks. Instead, OCC validates transactions before they commit to ensure no conflicts occurred during execution.

How It Works:

  • Execution Phase: Transactions execute without locking data items.
  • Validation Phase: Before committing, each transaction checks for conflicts.
  • Commit or Rollback: If no conflicts are found, the transaction commits. Otherwise, it is rolled back.

OCC is suitable for environments with low conflict rates, as it reduces the need for locking and improves performance.

Example:

In a Reservation System, OCC allows multiple users to select seats simultaneously, checking for conflicts only when a user tries to finalize the booking. If a conflict is detected, the system notifies the user to choose a different seat.


Comparison of Concurrency Control Techniques

Technique Description Example Use Case
Lock-Based Protocols Uses locks to control access and prevent conflicts. Online shopping cart with shared and exclusive locks.
Timestamp-Based Protocols Assigns timestamps to maintain execution order. Library system where transactions follow timestamp order.
Optimistic Concurrency Control Assumes low conflict, validates before commit. Reservation systems with minimal conflict.
