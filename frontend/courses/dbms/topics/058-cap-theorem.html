CAP Theorem in Distributed Systems

The CAP Theorem is a principle in distributed computing that states a distributed database system can achieve only two of the following three properties simultaneously:

  • Consistency (C): All nodes in a distributed system return the same data for the same query.
  • Availability (A): Every request receives a response, even if some nodes are unavailable.
  • Partition Tolerance (P): The system continues to operate even if there is a network failure or communication loss between nodes.

Key Points of CAP Theorem

  • Trade-off Requirement: Systems can achieve at most two of the three properties simultaneously but not all three.
  • Distributed System Design: CAP theorem helps determine the design goals and limitations of distributed databases.

1. Consistency (C in CAP)

Consistency ensures that every read request receives the most recent write or an error, regardless of the node that processes it. In a consistent system, all nodes have the same data view.

Example

In a banking system, if a customer deposits money, any subsequent query to check the balance must reflect the updated amount immediately. Inconsistent responses, like different balances from different nodes, would violate consistency.

2. Availability (A in CAP)

Availability ensures that every request to the system receives a response, regardless of failures or partial system downtime. An available system might return a response even if it’s not fully up-to-date, as long as it can process requests.

Example

In an e-commerce website, availability is prioritized so users can always browse and add items to their cart, even if the inventory data might be slightly outdated due to network delays.

3. Partition Tolerance (P in CAP)

Partition Tolerance means the system can continue to operate even if there is a failure in the network, causing a partition (break) between nodes. This tolerance is essential in distributed systems where network issues can temporarily divide the system into isolated groups of nodes.

Example

In a cloud-based application, network failures or communication issues between data centers should not prevent the system from functioning. Partition tolerance allows different sections of the system to keep working independently.

Trade-offs in CAP

The CAP theorem implies that a distributed system can prioritize only two of the three properties at a time:

1. CP (Consistency + Partition Tolerance)

Prioritizes consistency and tolerance to network partitions, sacrificing availability during partition events.

Example

Many banking systems prioritize CP to ensure accurate data, even if it means temporary unavailability.

2. AP (Availability + Partition Tolerance)

Prioritizes availability and tolerance to network partitions, sacrificing immediate consistency.

Example

DNS (Domain Name System) prioritizes AP to provide name resolutions even if updates to records are slightly delayed.

3. CA (Consistency + Availability)

Prioritizes consistency and availability, sacrificing partition tolerance. However, CA systems are difficult to maintain in distributed systems because network partitions are always a risk.


IRCTC Example of CAP Theorem

The IRCTC system, which handles online railway reservations in India, faces unique challenges in balancing consistency, availability, and partition tolerance:

  • Consistency: IRCTC needs consistent data so users see the correct number of available seats when booking tickets. However, ensuring perfect consistency is challenging because of high user load and network partitions.
  • Availability: Users expect round-the-clock access to the IRCTC booking system. During peak hours or network issues, availability might come at the expense of showing slightly outdated information.
  • Partition Tolerance: With millions of users and servers across different locations, IRCTC must be resilient to network partitions. Even if some servers cannot communicate, the system should still process requests and update bookings as soon as the network is restored.

CAP Trade-off in IRCTC

During high-traffic events, IRCTC might prioritize Availability and Partition Tolerance (AP), allowing users to book tickets even if the seat availability count is not immediately consistent across all nodes. In other scenarios, IRCTC might aim for Consistency and Availability (CA), temporarily sacrificing partition tolerance during system upgrades or maintenance to ensure accurate booking data.

Summary Table of CAP Properties and Examples

Property Description Example Use Case
Consistency All nodes reflect the same data at all times Banking: Every transaction is accurately reflected
Availability System responds to requests despite failures E-commerce: Users can browse and add items despite delays
Partition Tolerance System operates despite network partitions Cloud Applications: System keeps working even with network issues
