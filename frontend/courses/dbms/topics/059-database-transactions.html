What are Database Transactions?

A Database Transaction is a sequence of operations performed as a single logical unit of work. A transaction groups multiple actions, such as insertions, updates, or deletions, that must be executed together to maintain data integrity. A transaction ensures that either all operations are successfully completed or none are, preserving database consistency.

Example:

Consider a Bank Transfer between two accounts:

  • Deduct $100 from Account A.
  • Add $100 to Account B.

Both actions must succeed or fail together to prevent any inconsistencies, such as money disappearing or being duplicated.

Characteristics of a Transaction

A transaction in a database must exhibit the ACID properties:

  • Atomicity: Ensures that all operations within a transaction are treated as a single unit. If any part of the transaction fails, the entire transaction is rolled back, and no changes are applied to the database.
  • Consistency: Ensures that a transaction brings the database from one consistent state to another, maintaining database rules and constraints.
  • Isolation: Ensures that the operations in a transaction are isolated from others. Intermediate steps in a transaction are not visible to other transactions until the transaction is complete.
  • Durability: Ensures that once a transaction is committed, its changes are permanent, even in the event of a system failure.

Transaction States

A transaction can be in several states throughout its lifecycle:

  • Active: The transaction is currently executing.
  • Partially Committed: All operations have completed successfully, but changes are not yet permanently stored.
  • Committed: The transaction has been successfully completed, and all changes are now permanent.
  • Failed: An error occurred during the transaction, leading to a rollback.
  • Aborted: The transaction has been rolled back due to failure, undoing any changes it made.

Transaction Management

Transaction Management involves tracking the execution of transactions to ensure ACID properties are maintained. Key responsibilities include starting, committing, rolling back, and logging transactions. A transaction manager coordinates these operations to ensure data consistency and reliability.

Key Functions:

  • Start Transaction: Begins a new transaction.
  • Commit Transaction: Confirms and permanently applies the transaction changes.
  • Rollback Transaction: Reverses all changes made by the transaction, restoring the database to its prior state in case of failure.

Transaction Concurrency Control

Concurrency Control manages simultaneous transactions in multi-user environments to prevent conflicts and ensure data integrity. Without concurrency control, two transactions could lead to issues such as dirty reads, lost updates, or inconsistent data.

Common Concurrency Control Methods:

  • Lock-Based Protocols: Locks data items to prevent conflicting access by multiple transactions.
  • Timestamp-Based Protocols: Assigns a timestamp to each transaction to determine the order of execution.
  • Optimistic Concurrency Control: Assumes minimal conflicts and checks for conflicts before committing.

Example:

In an e-commerce system, two users might attempt to purchase the last item in stock simultaneously. Concurrency control ensures only one transaction can complete successfully, preventing overselling.


Single-User vs. Multi-User Transactions

Single-User Transactions: In a single-user environment, only one transaction can be processed at a time. Concurrency control is unnecessary, simplifying transaction management.

Multi-User Transactions: In multi-user environments, multiple transactions occur simultaneously, requiring concurrency control to manage data access and ensure integrity.

Comparison

Type Description Concurrency Control
Single-User Processes one transaction at a time. Not required
Multi-User Processes multiple transactions concurrently. Required
