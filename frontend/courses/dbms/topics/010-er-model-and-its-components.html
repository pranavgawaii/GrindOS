Entity-Relationship Model (ER Model)

The Entity-Relationship Model (ER Model) is a popular data modeling approach that represents data as entities and their relationships. It visually maps out the database structure, making it easier for database designers to translate real-world data into a logical structure. The ER model is often represented by an ER diagram, which helps in designing, developing, and understanding the database at a high level.

1. Entities

In the ER Model, an entity represents a real-world object or concept. Each entity has properties, called attributes, that describe it. For example, in a university database, students, courses, and professors are entities.

Characteristics of Entities

  • Uniqueness: Each entity is distinguishable from others in the database.
  • Attributes: Each entity has attributes that define its properties.
  • Types of Entities: Entities can be classified into strong and weak entities (explained below).

Example of an Entity: Consider an entity named Student in a university database. It has attributes such as Student_ID, Name, Age, and Major.

2. Strong vs. Weak Entities

Entities in an ER model can be classified into strong and weak entities based on whether they depend on other entities for their existence.

Strong Entity

A strong entity is an independent entity that can exist without relying on other entities. It has a unique identifier, known as a primary key.

  • Characteristics: Independent, identified by its own primary key.

Example: In a library database, Book is a strong entity because each book can be identified by a unique ISBN.

Weak Entity

A weak entity cannot exist without a strong entity. It does not have a unique identifier on its own and relies on a foreign key and partial key to be uniquely identified. Weak entities are typically represented in ER diagrams with double rectangles.

  • Characteristics: Dependent on a strong entity, lacks a unique primary key, identified by a combination of foreign key and partial key.

Example: In a university database, a Dependent entity (representing dependents of employees, like children or spouses) would be a weak entity, relying on the Employee entity for unique identification.

Entity Type Primary Key Foreign Key
Employee Strong Employee_ID -
Dependent Weak (Employee_ID, Dependent_Name) Employee_ID

3. Attributes

Attributes define the properties of entities. Each entity has a set of attributes that provide specific information about it. Attributes can be of various types based on their nature and purpose.

Types of Attributes

  • Simple Attributes: Cannot be divided further (e.g., Age).
  • Composite Attributes: Can be divided into smaller sub-attributes (e.g., Address can have Street, City, Zip Code).
  • Derived Attributes: Calculated from other attributes (e.g., Age derived from Date_of_Birth).
  • Single-valued Attributes: Hold only one value (e.g., Employee_ID).
  • Multi-valued Attributes: Can hold multiple values (e.g., Phone_Numbers).

Example of Attributes in a Student Entity: In the Student entity, attributes might include:

Attribute Type Example
Name Simple John Doe
Address Composite 123 Main St, NY
Age Derived 20 (from DOB)
Phone_Numbers Multi-valued 555-1234, 555-5678

4. Relationships

A relationship in an ER model describes how entities are connected to each other. Relationships can be between two or more entities, and they are represented by diamonds in ER diagrams.

Types of Relationships

  • One-to-One (1:1): Each instance of one entity is related to a single instance of another entity.
    Example: Each student has a unique ID card, and each ID card is assigned to one student.
  • One-to-Many (1:M): One instance of an entity is related to multiple instances of another entity.
    Example: A professor can teach multiple courses, but each course is taught by only one professor.
  • Many-to-Many (M:M): Multiple instances of one entity relate to multiple instances of another entity.
    Example: Students enroll in multiple courses, and each course has multiple students.

Example of Relationships in a University Database

In a university database:

Relationship Type Entity 1 Entity 2 Example Description
One-to-One Student ID_Card Each student has one unique ID card
One-to-Many Professor Course A professor teaches multiple courses
Many-to-Many Student Course Students enroll in multiple courses

5. Total Participation in Entity Sets

Total participation indicates that every instance of an entity must participate in a relationship. It’s also known as mandatory participation. In ER diagrams, total participation is represented by a double line connecting the entity to the relationship.

  • Characteristics: Every instance of the entity is required to be related to an instance of another entity in the relationship.

Example: In a university, each course must have an assigned professor. Here, Course has total participation in the taught_by relationship with Professor, indicating that every course requires a professor.

Entity Relationship Total Participation
Course taught_by Yes
Patient assigned_to Yes

ER Diagram Example

Consider a simplified library management system ER model with entities like Book, Author, Member, and Loan.

Entities and Attributes

  • Book: Attributes - Book_ID, Title, Author_ID.
  • Author: Attributes - Author_ID, Name.
  • Member: Attributes - Member_ID, Name.
  • Loan: Weak entity, representing loans of books by members. Attributes - Loan_ID, DueDate, Member_ID, Book_ID.

Relationships

  • Written_By: Book and Author have a one-to-many relationship (one author can write multiple books).
  • Borrowed_By: Loan and Member have a many-to-one relationship (one member can borrow multiple books).
