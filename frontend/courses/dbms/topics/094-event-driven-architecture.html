What is Event-Driven Architecture (EDA)?

Event-Driven Architecture (EDA) is a software design pattern in which actions or processes are triggered by the occurrence of events. Events are state changes or updates that are detected and processed asynchronously. EDA is often used in systems requiring responsiveness, scalability, and real-time processing.

Real-Life Relation

Online payment gateways use EDA to trigger notifications when a transaction is successful, sending alerts to both the merchant and the user in real-time.

Producers and Consumers

Producers

Generate events, such as a user clicking a button, a database update, or an IoT sensor emitting data.

  • Examples: Web servers, database triggers, IoT devices.

Consumers

Listen for and process events, taking actions based on the event data.

  • Examples: Notification systems, analytics tools, or data pipelines.

Real-Life Relation

An e-commerce platform generates events (e.g., order placed) that a consumer system (e.g., order fulfillment) listens to and processes.

Event Channels (Event Streams)

Event channels (or streams) are intermediaries that facilitate communication between producers and consumers. They ensure events are transferred in a structured manner, often using tools like Apache Kafka or Amazon Kinesis.

Types of Event Channels

  • Point-to-Point Channels: Events are sent from one producer to one consumer.
  • Publish-Subscribe Channels: Multiple consumers subscribe to a producer's events.

Real-Life Relation

Social media platforms use event streams to distribute live updates (e.g., comments or likes) to all relevant users in real time.

Event-Driven Triggers in DBMS

Event-driven triggers in a database management system (DBMS) are automated procedures executed in response to specific database events, such as INSERT, UPDATE, or DELETE operations.

Examples

  • After-Insert Trigger: Sends a notification when a new record is added to a table.
  • Before-Update Trigger: Logs changes before a record is modified.

Real-Life Relation

A banking system may use a trigger to monitor withdrawals and alert the user if the account balance falls below a threshold.

Asynchronous Communication

Asynchronous communication allows producers and consumers to operate independently, meaning the producer does not wait for the consumer to process an event. Events are queued in channels and processed when the consumer is ready.

Advantages

  • Decoupled Systems: Components operate independently.
  • Scalability: Multiple consumers can process events concurrently.
  • Fault Tolerance: Consumers can resume processing after downtime.

Real-Life Relation

Email systems process messages asynchronously, allowing users to send emails without waiting for them to be delivered instantly.
