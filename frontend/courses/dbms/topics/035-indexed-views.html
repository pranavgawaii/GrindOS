Indexed Views

An Indexed View (also known as a materialized view in some databases) is a view that physically stores the result set in the database with an index on it. Unlike standard views, which are virtual tables and pull data dynamically, indexed views maintain a stored copy of the data, making query retrieval faster.

Key Points of Indexed Views

  • Physically Stored Data: Indexed views store data with indexes, enhancing query performance.
  • Faster Access: Due to indexed storage, querying an indexed view is faster than querying a standard view.
  • Ideal for Aggregations: Indexed views are useful for scenarios involving complex joins or aggregations.

Standard Views vs. Indexed Views

Feature Standard View Indexed View
Data Storage Virtual table; data is not physically stored Stores data physically with indexes
Performance Slower; recalculates data each time it’s accessed Faster due to indexed storage
Use Case Frequently accessed but lightweight queries Heavy queries with joins, aggregations
Update Frequency Always reflects the latest data in underlying tables Requires periodic refresh for updated data

Example

Consider a Sales table:

Sale_ID Customer_ID Amount
1 101 200
2 102 150

A standard view on Sales dynamically retrieves data whenever queried, while an indexed view stores data in a materialized format, improving query performance for repeated use.

Creating Indexed Views

Creating an indexed view involves:

  1. Defining a view with a specific query.
  2. Adding a clustered index to the view, which materializes and stores the view data.

Syntax for Creating an Indexed View

CREATE VIEW view_name WITH SCHEMABINDING AS
SELECT columns FROM table_name WHERE condition;

CREATE UNIQUE CLUSTERED INDEX index_name ON view_name(column_name);

Example

To create an indexed view that shows total sales:

CREATE VIEW Sales_Summary WITH SCHEMABINDING AS
SELECT Customer_ID, SUM(Amount) AS Total_Sales
FROM dbo.Sales
GROUP BY Customer_ID;

CREATE UNIQUE CLUSTERED INDEX idx_sales_summary ON Sales_Summary(Customer_ID);

Result:

Customer_ID Total_Sales
101 200
102 150

Refreshing Materialized Views

Since indexed views store data physically, they must be refreshed to reflect updates in the underlying tables. Refreshing a materialized view can be done manually, set to refresh automatically at defined intervals, or triggered by specific actions.


Refresh Mechanisms

  • Automatic Refresh: Many databases provide options to refresh indexed views at specified intervals.
  • Manual Refresh: Refresh the view as needed using a specific command.
  • Trigger-Based Refresh: Set triggers on underlying tables to refresh the view when data changes.

Syntax for Refreshing Materialized Views (Example for Oracle)

EXEC DBMS_MVIEW.REFRESH('view_name');

Example

To refresh the Sales_Summary view after data changes in Sales:

EXEC DBMS_MVIEW.REFRESH('Sales_Summary');

This command updates the data in the Sales_Summary view to reflect recent changes in the Sales table.

Performance Benefits of Indexed Views

Indexed views provide significant performance benefits in environments with heavy data retrieval or complex queries:

  • Optimized Aggregations: Indexed views precompute complex aggregations, making retrieval faster.
  • Reduced Computation: Storing the computed data reduces the need for recalculating on each query.
  • Faster Joins: Indexed views can store join results, improving join performance.

Example

Consider querying Sales_Summary repeatedly to view total sales per customer. Without an indexed view, the database would aggregate the data each time. With an indexed view, it retrieves precomputed results, significantly reducing processing time.

Summary Table of Indexed View Features

Feature Description Example Usage
Creation Create a view with a unique clustered index CREATE VIEW ...; CREATE UNIQUE INDEX ...;
Data Storage Physically stores result set with indexes Ideal for repeated, data-heavy queries
Refresh Mechanisms Updates view to reflect table changes EXEC DBMS_MVIEW.REFRESH('view_name');
Refresh Intervals Scheduled or trigger-based refreshes Set refresh every 24 hours or on data change
Performance Benefit Faster retrieval for aggregated or joined data Total sales, summary views for large datasets
