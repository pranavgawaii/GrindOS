📂 Processes and CPU Scheduling
📄 Topic 3

1. What is Context Switching?

A Context Switch is the process of saving the current state (context) of an actively running process or thread so that it can be paused, and restoring the state of a ready process or thread to begin execution. This allows a single CPU core to run multiple processes concurrently.

2. The Role of the Dispatcher

The Dispatcher is the OS kernel module that takes control of the CPU after the CPU Scheduler has selected a process. The dispatcher's duties include:

  • Performing the context switch.
  • Switching the CPU from privileged Kernel Mode to non-privileged User Mode.
  • Jumping to the correct location in the user program (restoring the Program Counter) to resume execution.

Dispatch Latency: The time it takes for the dispatcher to stop one process and start another. This must be as small as possible.

3. Context Switch Overhead & Costs

Context switching is pure system overhead; the CPU executes zero user instructions during a context switch. The primary costs include:

  1. Register State Save/Restore: Saving general-purpose registers, Stack Pointer, and Program Counter to the PCB/TCB.
  2. TLB Flush (Process switch only): Switching virtual memory address spaces requires changing page tables, which invalidates the Translation Lookaside Buffer (TLB). This makes subsequent memory accesses slow.
  3. L1/L2 Cache Misses: The CPU cache is "cold" for the new process, resulting in memory latency overhead as data is fetched from slower main RAM.

🎯 Topic Q&A

Q1: What is Dispatch Latency?

Dispatch Latency is the time taken by the dispatcher to stop one running process, perform a context switch, change the CPU mode to User Mode, and load the next process to start running.

Q2: Why does process switching have higher overhead than thread switching?

Processes do not share address spaces. Switching processes requires changing the page table register (e.g., CR3 in x86), which invalidates and flushes the Translation Lookaside Buffer (TLB). Threads share memory, so thread switching retains the TLB mapping, avoiding this overhead.

Q3: When does a context switch happen?

A context switch occurs during a system call trap, a hardware interrupt (like a timer tick in preemptive scheduling), an explicit I/O request, or when a process voluntary terminates.

Q4: Is context switching done by hardware or software?

It is a cooperative effort. The CPU provides instructions to save/restore registers rapidly, but the OS scheduler and dispatcher code execute the logic to swap PCB pointers and manage transitions.

Q5: What is a "cold" cache in context switching?

When a new process is loaded, the CPU's high-speed L1/L2 caches contain memory blocks from the old process. The new process experiences cache misses and executes slowly until its active memory blocks are loaded into the cache (making it "warm").
