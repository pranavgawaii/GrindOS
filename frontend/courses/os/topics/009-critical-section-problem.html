1. What is the Critical Section Problem?

A Critical Section (CS) is a segment of code where a process accesses and modifies shared resources (like global variables, files, or buffers). A **Race Condition** occurs if multiple threads access and modify this shared state concurrently, causing data corruption.

The **Critical Section Problem** is to design a protocol ensuring that only one process executes inside its critical section at any given time.

2. Three Core Architectural Requirements

Any valid solution to the critical section problem must satisfy all three constraints:

  1. Mutual Exclusion: If process Pi is executing in its critical section, no other processes can execute in their critical sections for the same shared resources.
  2. Progress: If no process is executing in its critical section and some processes wish to enter, only those processes that are not executing in their remainder sections can participate in deciding which process enters next. This decision cannot be postponed indefinitely.
  3. Bounded Waiting: There must be a limit on the number of times other processes are allowed to enter their critical sections after a process has made a request to enter and before that request is granted. This prevents process starvation.

3. Software Solution: Peterson's Solution

A classic software-based solution for two processes (P0 and P1) sharing synchronization variables:

int turn;
boolean flag[2]; // Initialized to false

// Process P_i
void process_i() {
    int j = 1 - i;
    while(1) {
        flag[i] = true;        // Signal intent to enter
        turn = j;              // Give turn to other process
        
        while (flag[j] && turn == j); // Busy wait (spin lock)
        
        // ---- CRITICAL SECTION ----
        // Modify shared resource
        
        flag[i] = false;       // Reset intent
        
        // ---- REMAINDER SECTION ----
    }
}

Proof of Correctness

  • Mutual Exclusion: If both try to enter, `turn` can only be `0` or `1`. If `turn == j`, then Pi spins, guaranteeing only one process enters.
  • Progress & Bounded Waiting: When Pi exits, it sets `flag[i] = false`, letting the waiting process break its spin lock instantly.

🎯 Topic Q&A

Q1: What are the three requirements for a valid solution to the Critical Section problem?

They are: 1) Mutual Exclusion (only one process in CS). 2) Progress (processes waiting to enter decide who goes next). 3) Bounded Waiting (no process waits indefinitely/starves).

Q2: Why does Peterson's Solution fail on modern CPU architectures?

Peterson's solution assumes sequential consistency. Modern CPUs perform out-of-order execution and aggressive compiler optimizations (register caching), which can reorder the execution of flag and turn statements, violating mutual exclusion unless explicit **Memory Barriers** are used.

Q3: What is a Race Condition?

A race condition is an undesirable state where multiple concurrent threads read and write to a shared memory block, and the final state of the data depends entirely on the timing and scheduling of the threads.

Q4: Explain busy waiting (spin lock) and its disadvantage.

Busy waiting occurs when a thread repeatedly checks a condition in a loop (like `while(turn == j)`) to acquire a lock. This wastes CPU cycles and battery power, especially in single-core systems.

Q5: What are hardware atomic instructions?

Hardware atomic instructions are special CPU operations (like `TestAndSet` or `CompareAndSwap`) that execute read-modify-write instructions in a single, uninterruptible clock cycle at the hardware bus level.
