1. Mutex (Mutual Exclusion Lock)

A Mutex is a locking mechanism used to synchronize access to a critical resource. It is an ownership-based structure: only the thread that locked the mutex is authorized to unlock it.

  • State: Binary (Locked/Unlocked).
  • Use case: Simple mutual exclusion on variables or files.

2. Semaphore

A Semaphore is a signaling mechanism based on an integer value, accessed via two atomic operations: wait() and signal().

void wait(Semaphore S) {
    S.value--;
    if (S.value < 0) {
        block();
    }
}
void signal(Semaphore S) {
    S.value++;
    if (S.value ≤ 0) {
        wakeup();
    }
}
  • Binary Semaphore: Value is restricted to 0 or 1. Behaves like a mutex but has no ownership constraint.
  • Counting Semaphore: Value can range over a set, used to allocate pools of identical resources.

3. Monitor

A Monitor is an abstract, high-level programming language construct that guarantees mutual exclusion natively. Only one thread can be active inside a monitor block at any time. It uses **Condition Variables** (`wait()` and `signal()`) for synchronization.

4. Comparison Table

Feature Mutex Semaphore Monitor
Level Low-level OS locking Low-level OS signaling High-level Language construct
Ownership Yes (locking thread owns lock) No (any thread can signal) Implicit compiler control

🎯 Topic Q&A

Q1: What is the differences in ownership between a Mutex and a Semaphore?

A Mutex has a strict ownership constraint: only the thread that acquired the lock can unlock it. A Semaphore has no ownership; any thread can execute signal() to wake up a waiting thread, making it a signaling mechanism.

Q2: How does a counting semaphore manage resource allocation?

A counting semaphore is initialized to the number of available resources (e.g., 5). When a process requests a resource, it calls wait(), decrementing the count. When finished, it calls signal(), incrementing the count. If the count falls below 0, subsequent requests block until a resource is returned.

Q3: What is a Monitor in synchronization?

A monitor is a high-level language package/object that encapsulates shared variables and functions, automatically guaranteeing that only one thread can execute inside its functions at any time, protecting data from concurrent access.

Q4: What is the difference between busy-waiting and blocking semaphores?

A busy-waiting semaphore spins in a loop on wait(), wasting CPU cycles. A blocking semaphore suspends the thread, puts it into a queue, and transitions it to the waiting state, allowing other threads to run on the CPU.

Q5: Can you write a Python thread-safe counter lock example? 
import threading

lock = threading.Lock()
counter = 0

def increment():
    global counter
    with lock: # Mutex-lock equivalent
        counter += 1
