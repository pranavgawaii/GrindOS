1. Concept of Virtual Memory

Virtual Memory is a memory management technique that decouples the programmer's logical view of memory from physical RAM. This allows processes to execute even if they are only partially loaded in RAM, enabling applications larger than physical memory to run smoothly.

2. Demand Paging

Under a demand paging architecture, pages are loaded into physical frames only when they are actively referenced during execution (lazy swapper/pager). Each page table entry uses a Valid-Invalid (V/I) bit:

  • Valid (V): The page is currently loaded in RAM.
  • Invalid (I): The page is not in RAM (it is stored on secondary swap disk).

3. Page Fault Handling

When the CPU attempts to access an address marked as **Invalid** in the page table, it triggers a hardware exception called a Page Fault. The OS handles this via these structured steps:

  1. The OS traps the execution, checking an internal process control table to verify if the memory reference is valid.
  2. If the reference is invalid, the process is terminated (Segmentation Fault). If valid, we must retrieve the page.
  3. The OS locates a free frame in physical memory (if no frames are free, we run a **Page Replacement Algorithm**).
  4. The OS schedules a disk I/O operation to read the desired page from secondary swap disk into the allocated free frame.
  5. Once read is complete, the OS updates the page table: changes the page entry frame mapping and sets the status bit to Valid.
  6. The OS restarts the CPU instruction that was interrupted by the page fault, resuming execution seamlessly.

4. Thrashing and the Working Set Model

Thrashing occurs when a virtual memory system spends more time page-faulting and swapping pages in/out of secondary storage than executing actual instructions.

  • Cause: Happens when the collective working set sizes of all running processes exceed the available physical RAM.
  • Working Set Model: The OS tracks the set of pages active in a sliding time window (Δ). If the sum of all working sets (D) exceeds physical RAM (m), the OS suspends a process, preventing thrashing.

🎯 Topic Q&A

Q1: What is a Page Fault?

A page fault is a hardware interrupt triggered by the Memory Management Unit (MMU) when a running program attempts to access a virtual page that is not currently loaded in physical RAM.

Q2: How does the OS handle page faults when physical memory is completely full?

The OS executes a Page Replacement Algorithm (like LRU) to select a victim frame. If the victim's dirty bit is set, it writes the page back to disk. It then marks the victim's page table entry as Invalid, loads the new page into that frame, marks it as Valid, and restarts the instruction.

Q3: What causes memory thrashing?

Thrashing is caused by over-committing memory. If the degree of multiprogramming is too high, the OS allocates fewer frames than a process's active working set requires, causing continuous page faults and disk-swapping bottlenecks.

Q4: Explain temporal and spatial locality.

Temporal locality states that memory locations accessed recently are likely to be accessed again soon (e.g., loops). Spatial locality states that memory adjacent to recently accessed locations is likely to be accessed soon (e.g., arrays).

Q5: What is the Working Set Window delta in thrashing prevention?

The working set window delta is a configurable parameter representing the sliding window of recent page references used to determine which pages are actively needed by a process, allowing the OS to calculate total system frame demand.
