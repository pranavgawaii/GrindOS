📂 Processes and CPU Scheduling
📄 Topic 1
…

1. Process: The Active Entity

A Process is an execution instance of a program in memory. It is an active entity that requires system resources like CPU time, memory space (code, data, heap, stack), and file descriptors to run. Unlike a program, which is a passive entity stored on disk (e.g., an executable file), a process is dynamic and currently running.

Key Memory Segments of a Process:
  • Text Segment: Contains the compiled machine instructions (read-only).
  • Data Segment: Stores global and static variables.
  • Heap Segment: Dynamically allocated memory during runtime (malloc/new). Grows upwards.
  • Stack Segment: Temporary data storage for function call stacks, local variables, and return addresses. Grows downwards.

2. Thread: The Unit of Execution

A Thread (or Lightweight Process - LWP) is the smallest unit of CPU execution inside a process. Multiple threads belong to a single process, sharing the same address space (code, data, heap) and operating system resources, but each thread maintains its own independent registers, Program Counter (PC), and stack.

3. Process Control Block (PCB)

The OS manages processes using a kernel structure called the Process Control Block (PCB). The PCB acts as the process's identification card, containing all information needed to pause, schedule, and resume its execution.

PCB Element Description
Process ID (PID) Unique integer identifier assigned by the OS.
Process State Current state (New, Ready, Running, Waiting, Terminated).
Program Counter (PC) Address of the next instruction to be executed.
Registers State of CPU registers (Accumulators, Stack Pointers, General Purpose registers).
CPU Scheduling Info Process priority, pointer to scheduling queues.
Memory Management Info Page tables, segment tables, base and limit registers.
I/O Status List of open files, allocated network sockets, outstanding I/O requests.

4. Process State Transitions

A process moves through various states during its lifetime:

  • New: The process is being created but is not yet loaded into RAM.
  • Ready: Loaded in memory and waiting for the CPU scheduler to allocate processor time.
  • Running: Instructions are actively being executed by the CPU.
  • Waiting / Blocked: Waiting for an external event (like I/O completion or signal).

5. Context Switching

A Context Switch is the mechanism of saving the current CPU state (context) of a running process or thread into its PCB or TCB, and loading the saved state of a ready process/thread into CPU registers to start execution.

  • Process Context Switch: Involves switching virtual memory maps (changing Page Directory base registers, flushing the Translation Lookaside Buffer - TLB). This carries a high performance overhead.
  • Thread Context Switch: Since threads share the same page tables, a thread context switch does not require changing virtual memory maps or flushing the TLB. Only general-purpose registers, program counters, and stack pointers are switched. This is extremely fast.

🎯 Topic Q&A

Q1: What is the fundamental difference between a process and a thread?

A process is an isolated, independent unit of execution with its own dedicated memory space (code, data, heap, stack). A thread is a lightweight unit of execution within a process; sibling threads share the parent process's memory (code, data, heap) but maintain their own PC, registers, and stack.

Q2: Why is thread context switching faster than process context switching?

Process context switching requires the OS to switch virtual memory page tables and flush the Translation Lookaside Buffer (TLB), which is a heavy hardware operation. Threads share the same address space, so thread switching does not require changing page tables or flushing the TLB, keeping CPU cache hot.

Q3: Explain the role of the Process Control Block (PCB).

The PCB is a kernel data structure that stores all metadata needed to manage a process, including PID, process state, program counter, CPU registers, scheduling priority, memory allocation, and list of open files.

Q4: What is the difference between a program and a process?

A program is a passive entity stored as an executable file on secondary disk storage (e.g., .exe file). A process is an active entity loaded into physical RAM that is currently executing instructions on the CPU.

Q5: What is a zombie process and how is it resolved?

A zombie process is a process that has completed execution (via exit()) but still has an entry in the process table because its parent has not yet read its exit status via wait(). It is resolved when the parent executes wait(), or if the parent dies, the orphan is adopted by init/systemd (PID 1), which reaps it.
