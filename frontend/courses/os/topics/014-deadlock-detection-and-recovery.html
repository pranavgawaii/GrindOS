1. Deadlock Detection

If a system does not implement deadlock prevention or avoidance, deadlocks can occur. In this case, the operating system must run background **Deadlock Detection** algorithms to identify active circular wait blocks.

Wait-For Graph (WFG)

For systems where all resources have only single instances, the Resource Allocation Graph (RAG) is simplified into a **Wait-For Graph (WFG)** by removing resource nodes and drawing direct edges from Pi → Pj if process Pi is waiting for resource held by Pj.

  • The OS periodically runs cycle-detection algorithms (like Depth-First Search) on the Wait-For Graph.
  • Cycle Rule: If a cycle is detected in a single-instance Wait-For Graph, a deadlock exists.

2. Deadlock Recovery Methods

Once a deadlock is detected, the OS recovers using two main strategies:

A. Process Termination

  • Abort all deadlocked processes: Guaranteed to break the deadlock, but extremely expensive as all computational progress is lost.
  • Abort processes one-by-one: Abort one process, re-run the detection algorithm, and repeat until the deadlock is resolved. (High CPU overhead).

B. Resource Preemption

Forcibly reclaim a resource from one process and allocate it to another:

  1. Selecting a Victim: Choose the process that is cheap to abort or holds the most resources.
  2. Rollback: Return the victim process to a safe checkpoint state, resuming execution from that point.
  3. Starvation Prevention: Ensure a process isn't repeatedly selected as a victim by incorporating a "preemption counter" in selection logic.

🎯 Topic Q&A

Q1: What is a Wait-For Graph?

A Wait-For Graph is a simplified directed graph used for deadlock detection in systems with single-instance resources, created by removing resource nodes from the RAG and mapping direct edges from waiting process to holding process.

Q2: How does the OS detect a deadlock if resources have multiple instances?

The OS executes an algorithm similar to the Banker's safety check. It assumes all unfinished processes will request their remaining resources, simulates completion, and if any processes remain uncompleted, those processes are deadlocked.

Q3: How does process termination recover a system from deadlock?

By aborting one or more blocked processes, forcing them to release their allocated resources back to the pool, thereby breaking the circular wait chain.

Q4: Explain resource preemption rollback in deadlock recovery.

Rollback is the mechanism of returning a process to a previously saved checkpoint state in its execution history, letting the OS reclaim its resources temporarily without completely aborting the process.

Q5: What is the ostrich algorithm in deadlock handling?

The ostrich algorithm is a strategy of ignoring deadlocks completely (putting your head in the sand). It is used when deadlocks are extremely rare and the cost of prevention, avoidance, or detection is too high (used in general-purpose OSs like Windows and Linux).
