1. Multilevel Queue Scheduling

In real-world operating systems, processes have distinct requirements. Instead of maintaining a single ready queue, Multilevel Queue Scheduling partitions the ready queue into several separate queues based on process properties (e.g., system processes, interactive foreground jobs, background batch jobs).

  • Each queue has its own unique scheduling algorithm (e.g., Round Robin for interactive jobs, FCFS for background batch jobs).
  • Static Assignment: A process is assigned to a specific queue upon creation and cannot move between queues.
  • Inter-Queue Scheduling: Typically implemented as a strict preemptive priority scheduling (e.g., service the system queue first, then interactive, then batch). This can lead to starvation of background queues.

2. Multilevel Feedback Queue (MLFQ)

To eliminate the static mapping limitation and prevent starvation, the Multilevel Feedback Queue (MLFQ) was introduced. It allows processes to move dynamically between queues based on their execution history.

Core Rules of MLFQ

  1. Aging: If a process spends too much time waiting in a low-priority queue, the OS promotes it to a higher-priority queue, preventing starvation.
  2. Preemption: A process that uses its entire time quantum on the CPU is demoted to a lower-priority queue (penalizing CPU-bound processes).
  3. I/O Release: A process that voluntary yields the CPU before its quantum expires (I/O-bound process) is kept in the high-priority queue, ensuring high responsiveness.

🎯 Topic Q&A

Q1: What is the main difference between Multilevel Queue and Multilevel Feedback Queue?

In Multilevel Queue scheduling, processes are statically assigned to a queue and cannot move. In Multilevel Feedback Queue scheduling, processes can move dynamically between queues to optimize system performance and prevent starvation.

Q2: How does MLFQ identify if a process is CPU-bound or I/O-bound?

MLFQ monitors CPU usage. If a process uses its entire time quantum without blocking, the scheduler flags it as CPU-bound and demotes it. If it voluntary yields the CPU before the quantum ends for I/O, it remains in the high-priority queue.

Q3: How does MLFQ prevent starvation of batch background processes?

It uses Aging. If a background process waits in the low-priority queue for a set threshold, it is promoted to a higher-priority queue to guarantee CPU cycles.

Q4: Why do we use different scheduling algorithms for foreground and background queues?

Foreground queues require fast response times for interactive user actions (best served by preemptive Round Robin). Background queues handle long processing batch jobs where latency doesn't matter, making low-overhead FCFS more efficient.

Q5: Can you write a pseudo-implementation outline of queue promotion in Python? 
# Dynamic queue aging tick
def check_aging(queues, max_wait_time):
    for q_idx in range(1, len(queues)): # Skip highest queue
        for process in list(queues[q_idx]):
            if process.wait_time > max_wait_time:
                queues[q_idx].remove(process)
                queues[q_idx - 1].append(process) # Promote
                process.wait_time = 0
