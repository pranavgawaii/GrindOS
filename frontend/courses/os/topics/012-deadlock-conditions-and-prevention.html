1. Four Necessary Conditions for Deadlock

A deadlock occurs in an operating system if and only if all four of the following Coffman Conditions hold simultaneously:

  1. Mutual Exclusion: At least one resource must be held in a non-shareable mode (only one process can use it at a time).
  2. Hold and Wait: A process must be holding at least one resource and waiting to acquire additional resources held by other processes.
  3. No Preemption: Resources cannot be forcefully taken from a process; they must be released voluntarily.
  4. Circular Wait: A closed loop of processes exists where each process holds resources required by the next (e.g., P0 waits for P1, P1 waits for P0).

2. Deadlock Prevention

We can prevent deadlocks by designing system protocols that mathematically **negate** at least one of the four necessary conditions:

  • Negating Mutual Exclusion: Make all resources shareable (e.g., read-only files). However, this is impossible for non-shareable resources like printers or writes to database tables.
  • Negating Hold and Wait: Require processes to request and be allocated all their required resources at once before starting execution (low resource utilization).
  • Negating No Preemption: If a process holding resources requests another resource that cannot be immediately allocated, all its currently held resources are pre-empted and released.
  • Negating Circular Wait (Most Common): Establish a strict global ordering of all resource types, and force all processes to request resources in a strictly increasing numerical order.

🎯 Topic Q&A

Q1: What are the four Coffman conditions for deadlocks?

They are: 1) Mutual Exclusion. 2) Hold and Wait. 3) No Preemption. 4) Circular Wait. All four must hold simultaneously for a deadlock to exist.

Q2: How does global resource ordering prevent circular wait?

If each resource is assigned an integer index (e.g., R1=1, R2=2), and processes can only request a resource if its index is greater than all currently held resources, a closed wait loop cannot form because a process cannot wait for a lower-index resource held by another.

Q3: Why is negating the Mutual Exclusion condition difficult?

Because some hardware devices (printers) or transactional operations (writes to file/DB) are intrinsically non-shareable and require exclusive access to prevent race conditions.

Q4: Explain the difference between deadlock prevention and deadlock avoidance.

Deadlock prevention changes system design rules so deadlocks are structurally impossible (e.g., resource ordering). Deadlock avoidance dynamically checks if a resource allocation request leaves the system in a safe state before granting it (e.g., Banker's Algorithm).

Q5: What is Starvation vs. Deadlock?

A deadlock is a mutual, permanent freeze where processes are blocked waiting for each other. Starvation is a state where a process waits indefinitely because other processes are constantly prioritized, but the process is not blocked in a loop.
